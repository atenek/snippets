# Конвенция логирования filesync

Правила использования логирования во всём наборе `filesync` (core + get/diff/put).
Цель — единый, предсказуемый и недорогой по сопровождению механизм диагностики и
аудита. Спецификация на реализацию — [../todo/logging.md](../todo/logging.md).

> Коротко: **`logging` из stdlib + `dictConfig`**; конфигурируется **только точка
> входа** (CLI), библиотека только пишет; логи идут в **stderr**, человекочитаемый
> отчёт остаётся в **stdout**; уровень задаётся ключом `--logging`.

---

## 1. Принципы

1. **Стандарт, без зависимостей.** Только stdlib `logging` (+ `logging.config.dictConfig`).
   У набора нет сторонних зависимостей — логирование их не вносит.
2. **Библиотека не настраивает логирование.** Модули (`core`, `getfile`, …) только
   *получают* логгер и *пишут*. Хендлеры, уровни, формат — назначает **исключительно
   точка входа** (`main()` утилиты). Это правило «библиотека молчит, приложение решает».
3. **Отчёт ≠ лог.** Итоговая сводка (`_print_summary`) и любой машинный вывод — это
   **результат работы**, он идёт в **stdout** через `print`. Логи — это **диагностика
   и аудит**, они идут в **stderr** через `logging`. Две разные роли, два разных потока.
4. **Дёшево, когда выключено.** Сообщения собираются лениво (см. §6) — отключённый
   уровень не должен ничего стоить, даже на дереве в десятки тысяч объектов.
5. **Безопасно для произвольных имён.** Пути на Linux — произвольные байты; логирование
   не должно падать на не-UTF8 именах (см. §6).

---

## 2. Иерархия логгеров и дисциплина библиотеки

- Корневой логгер набора — **`filesync`**. Все сообщения идут через него и его детей.
- В каждом модуле объявляется логгер с **явным** именем-потомком:

  ```python
  import logging
  log = logging.getLogger("filesync.core")     # в core.py
  log = logging.getLogger("filesync.getfile")  # в getfile.py
  ```

  **Почему явное имя, а не `__name__`.** Утилиты запускаются двумя способами
  (`python -m filesync get …` и `python filesync/getfile.py …`); при втором `__name__`
  становится `__main__`/`core` и логгер выпадает из-под родителя `filesync`. Явное
  `"filesync.<модуль>"` делает иерархию стабильной независимо от способа запуска.

- В `filesync/__init__.py` к корневому логгеру набора подключается `NullHandler`:

  ```python
  logging.getLogger("filesync").addHandler(logging.NullHandler())
  ```

  Это убирает предупреждение «No handlers could be found», когда `core` используется
  как библиотека (например, из тестов) без конфигурации логирования.

- **Запрещено** в библиотечных модулях: `logging.basicConfig(...)`, `dictConfig(...)`,
  добавление потоковых/файловых хендлеров на уровне импорта, вызовы на корневом
  логгере (`logging.info(...)` без имени).

## 3. Уровни и их семантика (что писать и куда относить)

Семантика — **накопительная**, как в стандартном `logging`: выбранный уровень
включает себя и всё, что выше по важности (INFO ⇒ INFO+WARNING+ERROR).

| Уровень | Что относим сюда | Примеры в наборе |
|---|---|---|
| **ERROR** | Срыв запуска и нарушения целостности **с нашей стороны**: то, что мы могли и должны были сделать, но не сделали. | нет/пустой filelist; ignore-файл не найден; `RemoteError` (ssh/find/timeout); `copy.status = failed`; локальная копия **отсутствует** после распаковки; **md5 не совпал**; пойманное исключение (`log.exception`). |
| **WARNING** | Деградация из-за **внешних ограничений источника**, при которой обработка продолжается. | `no_permission` (не прочитать даже под sudo); `not_found` (путь из списка отсутствует на remote); каталог не перечислить (`enum_denied`); ненулевой `tar` с непустым stderr; «прочие» строки stderr от `find`; ignore-паттерн с пробелом. |
| **INFO** *(default)* | **Факт сохранения** данных или метаданных для файлового объекта (f/d/l) и рамка сессии. По одной записи на объект. | старт сессии (host, число входов, каталог, транспорт, sudo); `saved f|d|l <path>` (контент или только метаданные); `excluded <path> (rule: …)`; финальные счётчики. |
| **DEBUG** | Подробности **шагов, проверок и решений** с указанием **причины**. | вход/выход стадий (expand/metadata/transfer/extract/verify) и их коды; по каждому пути — kept/excluded/pruned, его `rel` и какое правило сработало (или почему оставлен: явный вход / предок / нет совпадения); причина классификации `copy.status`; набор захваченных предков; сверка md5 local↔remote. |
| **NONE** | Логирование выключено (записи не эмитятся). | — |

Разделительный принцип ERROR↔WARNING: **ERROR** — «мы не справились / целостность
нарушена»; **WARNING** — «источник не дал (права/отсутствие/перечисление), но мы
честно это зафиксировали и пошли дальше». Это совпадает с трактовкой кодов возврата
(пропуски → код 2).

## 4. Соответствие ключу `--logging`

| Значение CLI | Порог логгера `filesync` | Примечание |
|---|---|---|
| `NONE` | логирование выключено | только `NullHandler`, записи не идут |
| `ERROR` | `ERROR` | ERROR + CRITICAL |
| `WARN` | `WARNING` | `WARN` — устаревший алиас, маппится в `WARNING` |
| `INFO` | `INFO` | **по умолчанию** |
| `DEBUG` | `DEBUG` | всё, включая шаги и решения |

Значения регистронезависимы. Дефолт — `INFO`.

## 5. Назначение, потоки, формат, время

- **Два хендлера:**
  - **файл** — полный аудит запуска на выбранном уровне (`--logging`), детальный формат;
  - **консоль (`stderr`)** — только проблемы, уровень `max(выбранный, WARNING)`: пер-объектные
    INFO не флудят консоль, оператор видит лишь `WARNING`+, весь аудит лежит в файле.
- **Каталог логов — `logs/` в корне проекта** (имя **захардкожено**, ключами не настраивается).
  На каждый запуск создаётся вложенный `logs/run<ГГГГ-ММ-ДД_ЧЧ-ММ-СС>.<мс>/` с файлом
  `<command>.log` (напр. `getfile.log`).
- **Отчёт ≠ лог.** Человекочитаемая сводка и любой конвейерный вывод → **`stdout`** (`print`).
  Логи — в файл и в `stderr`. Не смешивать.
- **Формат.**
  - консоль (краткий): `%(asctime)s %(levelname)s %(name)s: %(message)s`;
  - файл (детальный): добавляются `%(module)s:%(lineno)d`.
- **Время — UTC**, чтобы совпадать с метками сессии (`core.utc_now`, `session.json`):
  `logging.Formatter.converter = time.gmtime` (ставится при конфигурации).
- **Сообщение** — короткое, с парами `ключ=значение`; путь — последним и через `%r`/санитайзер (§6).
- **UTF-safe файл:** `FileHandler(encoding="utf-8", errors="backslashreplace")` — не падать на байт-именах.

## 6. Правила написания вызовов

1. **Ленивое форматирование — обязательно.** Аргументы передаём в логгер, не
   собираем строку заранее. Особенно в циклах по объектам (дерево бывает 10⁴–10⁵).

   ```python
   log.info("saved %s %r", tag, path)          # да
   log.info(f"saved {tag} {path!r}")            # нет — форматируется всегда
   ```

2. **Дорогой DEBUG — под охраной.** Если для сообщения нужна нетривиальная подготовка:

   ```python
   if log.isEnabledFor(logging.DEBUG):
       log.debug("rel=%r matched rule=%r", rel, pat)
   ```

3. **Исключения — `log.exception` / `exc_info`.** В `except` пишем трассу на уровне ERROR:

   ```python
   except core.RemoteError as exc:
       log.error("remote failed: %s", exc)      # сообщение оператору
       # либо log.exception(...) когда нужна трасса
   ```

4. **Безопасность путей/кодировки.** Имена приходят через `surrogateescape` и могут быть
   не-UTF8. Логируем путь через `%r` **или** прогоняем через санитайзер
   (`s.encode(errors="surrogateescape").decode(errors="backslashreplace")`), чтобы
   форматтер не падал `UnicodeEncodeError` на stderr.
5. **Без секретов и контента.** Не логируем содержимое файлов, переменные окружения,
   stdin-пейлоады команд. md5, пути, имена пользователей/групп, коды возврата — можно.
   На DEBUG допустим `argv` ssh и шаблоны команд, но **не** их потоковый ввод.

## 7. Где и как конфигурируется

- Конфигурацию строит **общая функция в `core`** —
  `core.configure_logging(level, *, command, log_root=None) -> str | None` — которую
  **вызывает только `main()`** утилиты, один раз, до `run()`. Функция в core допустима:
  она не конфигурирует на импорте, а только когда её явно позвали из CLI. Возвращает
  каталог запуска `logs/run…/` (или `None` при `NONE`).
- Используется `logging.config.dictConfig` (`version: 1`, `disable_existing_loggers: False`),
  два хендлера: `file` (уровень = выбранный) и `console`→stderr (уровень `max(выбранный, WARNING)`).
- `NONE` → каталог не создаётся, хендлеры не подключаются (остаётся `NullHandler`), записи не идут.
- Каталог `logs` — **захардкожен**; `log_root` — внутренний параметр для тестов, **не** CLI-ключ.
- Все подкоманды (`get`/`diff`/`put`) принимают `--logging` и используют **одну и ту же**
  функцию — формат и поведение логов идентичны во всём наборе.

## 8. Чек-лист разработчика

- [ ] Логгер модуля — `logging.getLogger("filesync.<модуль>")` (явное имя).
- [ ] Никакой настройки логирования вне `main()`/`core.configure_logging`.
- [ ] Уровень события выбран по таблице §3 (ERROR/WARNING/INFO/DEBUG).
- [ ] Отчёт — в stdout `print`; диагностика — в логгер (stderr).
- [ ] Ленивые `%`-аргументы; дорогой DEBUG под `isEnabledFor`.
- [ ] Путь логируется безопасно (`%r`/санитайзер).
- [ ] Никаких секретов/контента в сообщениях.
