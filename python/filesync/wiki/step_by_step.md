# Реальный сценарий: забрать → отредактировать → накатить

Пошаговый разбор типовой задачи: пользователь снимает набор файлов с хоста,
правит **содержимое** одного файла и **права/группу/владельца** других, затем
накатывает результат на целевой хост — **без погружения в десятки настроек**.

Хост в примере — `192.168.234.129`. Все команды ниже реально проверены.

---

## 0. Главная идея (почему так)

Зеркало `files/` распаковывается на локальной машине **не от root**, поэтому
владелец и часть прав на нём **неавторитетны** (владелец = локальный пользователь).
Значит «поменять права/владельца» правкой самого зеркала нельзя. Решение —
разделить два редактируемых слоя:

| Что меняем | Где правим |
|---|---|
| **содержимое** файла | прямо в `files/` (raw-files-local) |
| **права / владелец / группа** | в `restore.txt` (одна строка на объект) |

Всё остальное (`*.tar`, `metadata/*.json`, `*.csv`) — **read-only** и не трогается.

---

## 1. Шаг 1 — забрать данные (get)

```bash
echo "/tmp/sbs_src" > files.txt
python -m filesync get files.txt 192.168.234.129 --suffix message -p ./backups/
```

Появляется сессия `backups/192.168.234.129__2026-06-03_07-23-08__message/`
(без `/` — одна папка `[<prefix>__]<host>__<дата>[__<suffix>]` в `--path`; со `/`
в `--prefix`/`--suffix` — дерево бэкапов, напр. `--prefix cluster/nodeA`):

```
192.168.234.129__2026-06-03_07-23-08__message/
├── 192.168.234.129__2026-06-03_07-23-08__message.tar   (r--r--r--)  payload — faithful, RO
├── files/                                               (rw)        raw-files-local — РЕДАКТИРУЕМО
├── restore.txt                                          (rw)        override атрибутов — РЕДАКТИРУЕМО
└── metadata/                                            (r--r--r--)  RO
    ├── manifest.json    manifest.csv    session.json
```

> Имя сессии продублировано в имени папки **и** имени `*.tar` — архив остаётся
> самоописательным, даже если его вынесли из папки.

`restore.txt` заранее заполнен **фактическими** значениями с хоста — пользователь
видит тип и текущее состояние, меняет только нужное:

```
# <type> <perms> <owner> <group> <remote_path>   (type: d=dir f=file l=symlink; '-' = как в metadata)
d 0775 atenek atenek /tmp/sbs_src
f 0644 atenek atenek /tmp/sbs_src/doc.txt
f 0644 atenek atenek /tmp/sbs_src/g.conf
f 0644 atenek atenek /tmp/sbs_src/o.conf
f 0600 atenek atenek /tmp/sbs_src/p.conf
```

Формат: **тип + три поля + путь-остаток** (путь может содержать пробелы; имена
пользователей/групп — нет, поэтому разбор однозначен). `type` — 1 символ
`d`/`f`/`l` (dir/file/symlink), информативный (тип при накате берётся из metadata).
`-` в поле = «как в metadata».

---

## 2. Шаг 2 — отредактировать локальную копию

Пользователь хочет: сменить **содержимое** `doc.txt`, **права** `p.conf`,
**группу** `g.conf`, **владельца** `o.conf`.

```bash
SESS=backups/192.168.234.129__2026-06-03_07-23-08__message

# (a) содержимое — правим файл прямо в зеркале
echo "EDITED LOCALLY by user" > "$SESS/files/tmp/sbs_src/doc.txt"

# (b)(c)(d) атрибуты — правим restore.txt
#   p.conf: 0600 -> 0640   |   g.conf: group atenek -> nogroup   |   o.conf: owner atenek -> nobody
```

`restore.txt` после правки:

```
d 0775 atenek atenek  /tmp/sbs_src
f 0644 atenek atenek  /tmp/sbs_src/doc.txt
f 0644 atenek nogroup /tmp/sbs_src/g.conf
f 0644 nobody atenek  /tmp/sbs_src/o.conf
f 0640 atenek atenek  /tmp/sbs_src/p.conf
```

Это **весь** объём действий пользователя: один файл с тремя полями + правка
содержимого нужных файлов. Никаких флагов на каждый атрибут.

---

## 3. Шаг 3 — накатить на хост (put)

```bash
# сначала всегда план (ничего не пишет)
python -m filesync put "$SESS" 192.168.234.129 --target-root /tmp/sbs_restore --dry-run

# применить
python -m filesync put "$SESS" 192.168.234.129 --target-root /tmp/sbs_restore --yes
```

Никаких доп. ключей: `put` **всегда** читает `restore.txt` сессии (выбор объектов
+ атрибуты) и берёт контент из `files/`. План показывает **итоговые** (с учётом
правок) атрибуты:

```
План:
  [create] 0775 atenek:atenek  /tmp/sbs_restore/tmp/sbs_src
  [create] 0644 atenek:atenek  /tmp/sbs_restore/tmp/sbs_src/doc.txt
  [create] 0644 atenek:nogroup /tmp/sbs_restore/tmp/sbs_src/g.conf
  [create] 0644 nobody:atenek  /tmp/sbs_restore/tmp/sbs_src/o.conf
  [create] 0640 atenek:atenek  /tmp/sbs_restore/tmp/sbs_src/p.conf
  -> create=5 overwrite=0 skip=0
```

Результат на хосте (проверено):

```
doc.txt  perms=644 owner=atenek group=atenek   content="EDITED LOCALLY by user"
p.conf   perms=640 owner=atenek group=atenek          ← права изменены
g.conf   perms=644 owner=atenek group=nogroup         ← группа изменена
o.conf   perms=644 owner=nobody group=atenek          ← владелец изменён
```
`Restore: OK 5/5`.

## 4. Правило (минимум магии)

`put <session>` всегда использует данные **папки сессии**:

| Что | Источник |
|---|---|
| **какие** объекты накатывать | строки в `restore.txt` (удалил строку → не накатывается) |
| **атрибуты** (perms/owner/group) | поля `restore.txt`; `-` → из metadata |
| **контент** | `files/` (правки подхватываются) |
| недостающие родители | `mkdir -p` нейтрально (атрибуты не трогаются) |

Пользователь правит только **два места** — `files/` (контент) и `restore.txt`
(выбор + атрибуты). Без правок выходит «точно как было». `payload.tar` остаётся
«холодным» архивом и put не используется.

---

## 5. diff — отдельная история

`diff` сравнивает локальную копию с хостом и тянет дельту. Он **опирается на
неизменяемые** `metadata` + `payload` (RO) и **не предполагает** правок:

```bash
python -m filesync diff "$SESS" 192.168.234.129 --suffix daily
```

Защита от случайной правки baseline: если файл метаданных оказался **не**
read-only, diff печатает предупреждение (защита не стопроцентная, но ловит
типичный случай):

```
warning: metadata is writable (not read-only), may have been modified — diff trusts it as-is: .../manifest.json
```

## 6. Анализ подхода (реализация + user-cases)

**Сильные стороны**
- Разделение «контент vs атрибуты» отражает реальность (зеркало не несёт
  авторитетных владельца/прав). Пользователь правит ровно два понятных места.
- `restore.txt` предзаполнен фактом → виден текущий стейт, меняешь только нужное;
  он же служит списком выбора (удалил строку → объект не накатывается).
- Единая форма `put <session>` без флагов режима — минимум развилок.
- Верификация осмысленна и после правок: put пересчитывает ожидаемый MD5 из
  `files/` и сверяет владельца/группу/права по итоговым значениям.
- Имена пользователей/групп (а не числа) — человеко-понятно; `chown` принимает
  имена; для безымянного uid (orphan) есть откат на число.
- Родительские каталоги get захватывает с реальными правами (в `restore.txt`);
  непереч исленные родители создаются нейтрально → системные каталоги не клоббятся.

**Ограничения / на что обратить внимание**
- `put` всегда берёт контент из `files/` → **перезаливается весь выбранный набор**
  (даже неизменённые файлы). Для больших наборов это лишний трафик — решается
  оптимизацией 1.
- Override по **именам** предполагает, что имя существует на целевом хосте.
  Для переноса на хост с другим маппингом имён это намеренно (имя важнее uid);
  кому нужен точный uid — пишет в restore.txt число.
- restore.txt не управляет mtime (берётся из metadata; есть `--no-set-times`).
- Устройства/FIFO/сокеты через put не накатываются — только вручную из `payload.tar`.

**Предложения по оптимизации / развитию**
1. **Attr-only fast path.** Если контент файла не менялся (MD5 `files/` совпал с
   manifest), брать контент из `payload.tar` (без перезаливки), а атрибуты
   применять отдельно. Экономит трафик на больших наборах. ТЗ:
   [opt_attr_fast_path.md](../todo/opt_attr_fast_path.md).
2. **Валидация restore.txt.** Предупреждать о строках с путями вне manifest и о
   некорректном окталном поле — раннее обнаружение опечаток.
3. **Подпись манифеста.** Для более надёжного контроля целостности metadata —
   хранить HMAC/контрольную сумму манифеста (сейчас — эвристика «RO/не RO»).

**Итоговая рекомендация:** оставить текущую модель (она закрывает заявленный
кейс с минимумом действий) и добавить оптимизацию №1 (attr-only fast path) и
№3 (валидацию) как следующий инкремент.

## 7. Шпаргалка артефактов сессии

| Артефакт | Режим | Назначение |
|---|---|---|
| `<host>__<date>__<msg>.tar` | RO `0444` | faithful payload (для точного restore) |
| `metadata/manifest.json` `.csv` `session.json` | RO `0444` | метаданные/аудит |
| `files/` | обычный | raw-files-local — правка **контента** |
| `restore.txt` | `0644` | правка **прав/владельца/группы** для put |
