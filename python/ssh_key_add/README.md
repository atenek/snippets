# add_ssh_key_multi_host.py

Раскладка одного SSH-публичного ключа по списку хостов при первичном
доступе **по паролю**. Без сторонних зависимостей — только стандартная
библиотека Python 3 и системный `ssh`.

## Что делает

Для каждого хоста из `hosts_list.txt`:

1. подключается по `ssh` с паролем (пароль подаётся через pty, т.к. `ssh`
   не читает его из обычного stdin);
2. идемпотентно дописывает `PUBLIC_KEY` в `~/.ssh/authorized_keys`
   пользователя (с корректными правами 700/600);
3. подтверждает результат маркером от удалённой стороны и печатает статус.

Повторный запуск безопасен: существующий ключ не дублируется.

## Использование

1. Вписать свой публичный ключ в константу `PUBLIC_KEY` в скрипте.
2. Создать `hosts_list.txt` рядом со скриптом, по одной записи на строку:

   ```
   # host user password
   10.0.0.5   admin   S3cretPass
   server01   deploy  Pa$$w0rd
   ```

   Пустые строки и строки на `#` игнорируются.
3. Запустить:

   ```
   python3 add_ssh_key_multi_host.py
   ```

Код возврата `0` — все хосты успешны; `1` — был хотя бы один сбой.

## Статусы

- `OK: ключ добавлен` — ключ записан и подтверждён на хосте;
- `OK: ключ уже был на хосте` — ничего менять не потребовалось;
- `СБОЙ (rc=...): ключ НЕ подтверждён` — соединение/аутентификация/таймаут;
  под строкой печатается вывод сессии для диагностики.

## Как это работает и почему так

Ключевые технические решения и причины, по которым они необходимы.

### Пароль подаётся через pty с управляющим терминалом

`ssh` принципиально читает пароль **не из stdin, а из управляющего
терминала** (`/dev/tty`) процесса. Поэтому недостаточно просто записать
пароль в stdin дочернего процесса — его нужно «напечатать» в терминал.

Скрипт:

1. создаёт пару pty (`pty.openpty()`) и подключает slave-конец к
   `stdin/stdout/stderr` процесса `ssh`;
2. в `preexec_fn` (между `fork` и `exec`) переводит процесс в новую сессию
   `os.setsid()` и делает pty его **управляющим терминалом** через
   `ioctl(TIOCSCTTY)`.

Без шага 2 у `ssh` нет управляющего tty: он пытается запустить графический
`ssh-askpass`, не находит его и падает с `Permission denied`. Дополнительно
из окружения убираются `SSH_ASKPASS`/`DISPLAY` и выставляется
`SSH_ASKPASS_REQUIRE=never`, чтобы `ssh` гарантированно спрашивал пароль в
нашем pty, а не во внешней программе.

### Приглашение пароля ищется во всём накопленном выводе

pty отдаёт данные произвольными кусками, и подстрока `password:` может
разорваться между двумя `os.read()`. Поэтому поиск приглашения идёт по всему
накопленному буферу, а не по отдельному куску — иначе пароль иногда не
отправлялся и сессия молча отваливалась по таймауту.

### Удалённая команда идемпотентна и самопроверяема

На хосте выполняется небольшой `sh`-скрипт под `set -eu` (любой сбой
подготовки прерывает выполнение, без ложного добавления ключа):

- `mkdir`/`chmod` приводят `~/.ssh` и `authorized_keys` к правам 700/600;
- если файл не заканчивается переводом строки — он дописывается, чтобы новый
  ключ не «склеился» с последней строкой;
- `grep -qxF` ищет **точное совпадение целой строки**: если ключ уже есть —
  печатается `KEY_ALREADY_PRESENT`, иначе ключ добавляется и печатается
  `KEY_ADDED_OK`.

Питон-сторона считает хост успешным **только при наличии маркера** в выводе,
а не по коду возврата `ssh`. Это и есть гарантия «ключ реально на месте».

### Таймауты и единственная попытка пароля

Используются `ConnectTimeout`, общий таймаут сессии и
`NumberOfPasswordPrompts=1` — при неверном пароле или недоступном хосте
скрипт не зависает, а помечает хост сбоем и идёт дальше.

## Безопасность

⚠️ `hosts_list.txt` содержит пароли открытым текстом. Держите файл с правами
`600` и удалите его сразу после первичной раскладки ключей:

```
chmod 600 hosts_list.txt
shred -u hosts_list.txt   # или rm
```
