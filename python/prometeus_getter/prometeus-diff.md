### prometeus-diff.py

Консольная утилита для **однократного** hot-сравнения Prometheus-метрик,
отдаваемых двумя endpoint'ами (например, двумя нодами одного балансировщика
или "до/после" изменения конфигурации). Оба endpoint'а опрашиваются
одновременно (asyncio + aiohttp), после чего утилита показывает, какие
метрики отличаются.

##### Многократный запуск 
Многократный запуск обеспечивается внешней утилитой, например watch:
Без `--log-datetime` (по умолчанию) каждый запуск создаёт **свой** файл лога с микросекундной меткой времени
```bash
watch -n 1 './.venv/bin/python prometeus-diff.py'
```

Многократный запуск и логирование в один файл (дописыванием) 
```bash
PROMETEUS_DIFF_START_TIME=$(date '+%Y-%m-%d_%Hh%Mm%S')
watch -n 1 './.venv/bin/python prometeus-diff.py --log-datetime "$PROMETEUS_DIFF_START_TIME"'
```


#### Запуск и указание конфигурации

По умолчанию настройки берутся из `prometeus-diff_conf.json` рядом со скриптом или задаются явно через ключ `--settings`.

```bash
./.venv/bin/python prometeus-diff.py
./.venv/bin/python prometeus-diff.py --settings /path/to/my_conf.json
```


#### Конфигурация

Пример `prometeus-diff_conf.json`:

```json
{
  "endpoint1": "http://192.168.0.1:62300/metrics",
  "endpoint2": "http://192.168.0.2:62300/metrics",
  "white_list": [],
  "black_list": [],
  "timeout": 5.0
}
```

| Поле | Обязательность | Назначение |
|------|-----------------|------------|
| `endpoint1` / `endpoint2` | да | полные URL опрашиваемых endpoint'ов |
| `white_list` | нет, по умолчанию `[]` | какие метрики **включить** в сравнение |
| `black_list` | нет, по умолчанию `[]` | какие метрики **исключить** из сравнения |
| `timeout` | нет, по умолчанию `5.0` | таймаут HTTP-запроса к каждому endpoint'у, в секундах |

#### white_list и black_list

Оба поля — списки шаблонов **регулярных выражений** (Python `re`, `re.fullmatch`). 
Шаблон должен совпасть с именем метрики **целиком**, поэтому для частичного совпадения нужно использовать `.*` явно:

- `"cpu_usage"` — совпадёт только с метрикой `cpu_usage`, но не с  `cpu_usage_total`;
- `"cpu_.*"` — совпадёт с любой метрикой, начинающейся на `cpu_`;
- `".*_total"` — совпадёт с любой метрикой, заканчивающейся на `_total`.

Сравнение идёт с именем метрики **без label'ов** — то есть c частью до первой `{`. 
Поэтому один шаблон в `black_list`/`white_list` маскирует / пропускает сразу все 
label-варианты метрики (`http_requests{code="200"}`, `http_requests{code="500"}` 
и т.д. — все они подпадают под `"http_requests"`).

Каждый список проверяется по порядку, до первого совпавшего шаблона — дальше по списку не идём (совпадение уже найдено).

### Как они работают вместе

1. Если `white_list` **непустой** — в сравнение попадают только метрики, совпавшие хотя бы с одним его шаблоном. 
   Остальные метрики в сравнении не участвуют (в summary — "замаскировано(white_list)").
2. Если `white_list` **пустой или отсутствует** — в сравнение изначально попадают вообще все метрики.
3. К тому, что осталось после шагов 1–2, применяется `black_list`: метрики, совпавшие хотя бы с одним его шаблоном, 
   из сравнения исключаются ("замаскировано(black_list)").

`white_list` — это "сузить область сравнения только до нужного", 
`black_list` — "убрать шум из того, что осталось". 
Совместно не конфликтуют, `black_list` всегда применяется поверх `white_list`.

## Примеры

**1. Сравнивать вообще всё** (оба списка пустые):

```json
{
  "endpoint1": "http://host1:9100/metrics",
  "endpoint2": "http://host2:9100/metrics",
  "white_list": [],
  "black_list": []
}
```

**2. Исключить заведомо "шумные" метрики** (счётчики аптайма, случайные id и т.п.), сравнивая всё остальное:

```json
{
  "endpoint1": "http://host1:9100/metrics",
  "endpoint2": "http://host2:9100/metrics",
  "black_list": ["process_start_time_seconds", ".*_uptime_seconds", "go_.*"]
}
```

**3. Сравнивать только конкретную группу метрик** (например, только всё, что относится к HTTP и BGP), остальное игнорировать:

```json
{
  "endpoint1": "http://host1:9100/metrics",
  "endpoint2": "http://host2:9100/metrics",
  "white_list": ["http_.*", "bgp_.*"]
}
```

**4. Сравнивать только группу метрик, но с исключением** (весь `http_*`, кроме внутренней технической метрики `http_internal_debug`):

```json
{
  "endpoint1": "http://host1:9100/metrics",
  "endpoint2": "http://host2:9100/metrics",
  "white_list": ["http_.*"],
  "black_list": ["http_internal_debug"]
}
```

## Формат вывода

Первая строка — summary: сколько метрик получено с каждого endpoint'а, сколько совпало, 
сколько отличается и сколько замаскировано `white_list` / `black_list`. 
Далее — по одной строке на каждую отличающуюся метрику: метрика, отсутствующая на одной из сторон, 
показывается с `-` вместо значения.

```
SUMMARY: total: ( 10  /  9 ); "eq":  0; "diff":  1; "match_white_list":  9; "filter_black_list":  0.
bgp_prefixes_received: endpoint1=161  endpoint2=-
http_requests{code="200",method="get"}: endpoint1=120  endpoint2=118
```

Значения сравниваются численно, если это возможно (`"42"` и `"42.0"` считаются равными), иначе — как строки.

Вывод одновременно печатается на экран и сохраняется в лог
`log/metrics_diff_<YYYY-MM-DD>_<HH>h<MM>m<SS>-<ms>.log` (свой файл на
каждый запуск), либо в `log/run_<--log-datetime>.log` (один файл на
серию запусков) — см. раздел "Лог при многократном запуске" выше.

Код возврата: 
`0` — различий нет, 
`1` — есть отличающиеся метрики,
`2` — ошибка конфигурации (например, некорректный regexp в `white_list`/ `black_list` или отсутствует обязательное поле).
