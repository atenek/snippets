


### 1. `port_info.ip` в `trex_cfg.yaml`

Это IP-адрес **физического порта TRex** — сетевого интерфейса, который реально существует в системе и участвует в ARP.

Когда TRex поднимается, он использует этот адрес чтобы **отвечать на ARP-запросы** от соседних устройств. Например, если LB хочет узнать MAC Port 0 TRex — он шлёт ARP "кто такой 192.168.10.10?", и Port 0 отвечает своим MAC.

Этот адрес также используется TRex для **разрешения MAC next-hop**: TRex делает ARP-запрос к `default_gw` и запоминает его MAC — именно этот MAC потом подставляется в `Ether(dst=...)` исходящих пакетов.

```
port_info.ip = "я есть вот этот адрес на этом порту"
               (нужен для ARP, для идентификации порта в сети)
```

### 2. `IP(src=...)` в профиле трафика

Это IP-адрес **внутри генерируемого пакета** — то, что TRex кладёт в поле Source IP заголовка IP генерируемого UDP-пакета.

TRex как генератор трафика может класть туда **абсолютно любой адрес** — он не проверяет, совпадает ли он с реальным IP порта. Это и есть суть traffic generator: подделывать адреса так, как нужно для теста.

```
IP(src=...) = "что написано в заголовке пакета"
              (нужен для того, чтобы RS знал, куда слать ответ)
```

## Почему я использовал один и тот же IP — и почему это неточно

В моём примере `192.168.10.10` стоит в обоих местах. Формально это работает, но создаёт ложное впечатление что это одна сущность. На практике это **два независимых решения**:

| | `port_info.ip` | `IP(src=...)` в пакете |
|---|---|---|
| Что это | IP физического порта TRex | IP клиента внутри пакета |
| Кто его видит | Сетевое оборудование (ARP) | LB и RealServer (IP-заголовок) |
| Можно ли их разделить | Да | Да |
| Зачем совпадают в примере | Упрощение | Упрощение |

**Корректный сценарий где они различаются:**

```
port_info[0].ip = 172.16.0.10     # реальный IP порта TRex в management-сети
                                   # нужен только для ARP с LB

IP(src="10.100.0.0/16")           # CLIENT_IP = диапазон эмулируемых клиентов
                                   # TRex может генерировать миллионы разных src IP
                                   # из этого диапазона через STLVmFlowVar
```

Так TRex эмулирует тысячи клиентов с разными IP, хотя физический порт имеет один конкретный адрес.

## Практический вывод для вашей схемы DSR

`port_info[1].ip` (Port 1) — это особенно важная сущность. RS будет слать ответы на `dst IP` = CLIENT_IP, то есть на тот адрес, который прописан в `IP(src=...)` профиля. Чтобы эти ответы физически пришли на Port 1 TRex, нужно чтобы маршрут до CLIENT_IP вёл именно туда. Если `port_info[1].ip` и `IP(src=...)` — разные подсети, маршрутизация ответов может не сработать.

Именно поэтому в простых тестовых стендах их делают одинаковыми — чтобы не настраивать отдельную маршрутизацию.
