Посмотреть **core dump** и получить **stack trace (стек вызовов)**, используется `gdb`. 

---

# 1. Открыть core dump

## Если обычный core-файл:

```bash
gdb ./test core
```

## Если используется systemd:

```bash
coredumpctl list
coredumpctl gdb <PID>
```

---

# 2. Посмотреть стек вызовов

Внутри `gdb`:

```gdb
bt
bt full # подробнее

```

Покажет:

* цепочку вызовов функций
* аргументы
* локальные переменные (если есть `-g`)

---

# 3. Пример вывода

```text
#0  crash_function () at test.c:10
#1  main () at test.c:20
```

Это означает:

* падение произошло в `crash_function`
* вызвана из `main`

---

# 4. Перемещение по стеку

```gdb
frame 0   # текущий (где crash)
frame 1   # выше по стеку
```

---

# 5. Посмотреть код

```gdb
list
```

или:

```gdb
list 10
```

---

# 6. Посмотреть переменные

```gdb
info locals
info args
```

или конкретно:

```gdb
print variable_name
```

---

# 7. Посмотреть причину падения

```gdb
info signal
```

Часто:

* `SIGSEGV` — обращение к неверной памяти
* `SIGABRT` — abort()

---

# 8. Если стек «плохой»

Если видишь что-то вроде:

```text
#0  0x0000000000401137 in ?? ()
```

значит:

* нет debug symbols → нужно `-g`
* или бинарник не совпадает с core dump

---

# 9. Полезные команды сразу

Можно сразу запустить:

```bash
gdb -ex "bt" -ex "quit" ./test core
```

---

# 10. Для systemd (быстро)

```bash
coredumpctl gdb
(gdb) bt
```

---

# Практический минимум

Если коротко:

```bash
gdb ./binary core
(gdb) bt
```

---

# Частые проблемы

* ❌ бинарник перекомпилирован после падения
* ❌ нет `-g`
* ❌ core dump от другой версии программы
* ❌ оптимизации (`-O2`) усложняют стек

---

Если хочешь, могу разобрать реальный core dump (вывод `bt full`) и помочь интерпретировать причину падения.
